# UX Factors

## 1. Learnability and Intuitive Use
**Definition**

The functionality of the product should be immediately apparent to users of similar products and applications. For new users, functionality should be able to be conveyed in a short time period, and knowledge of use should be easy to retain. The product should function in a wide range of use-cases, giving the user the feeling that the tool is flexible.

**Relevance for CS Students**

CS Students often have previous experience with AI Chatbots. There is an established design language across many AI chatbots; conforming to this design language allows these students to instantly pick up the app, increasing the adoption rate and ease of use. While many students have used chatbots, it cannot be assumed that all students have. To ensure that these users can have a good experience as well, the system being easy to understand and pick up will enable these students to quickly master the tool without additional burden.

**Concrete Example Interaction**

When a student first signs into ESBot, they must identify how to request an explanation. When the input field is obvious and placed in an intuitive position, the user can much more rapidly begin their work.


## 2. Clarity and Comprehensibility
**Definition**

Questions, answers, and feedback provided by the product should be clear and effectively provide information in a pedagogical manner. All information provided should be in alignment with the student’s learning materials and grounded in reality.

**Relevance for CS Students**

Students are tasked with learning a large amount and variety of information. Being forced to sift through false or poorly presented information would cause many users to not return to the app.

**Concrete Example Interaction**

When a user asks for quiz questions about boolean logic, they expect questions that pertain to their course material and are well-phrased. Meeting this expectation will lead to more user satisfaction.


## 3. Information Efficiency
**Definition**

Information should be communicated concisely, without extraneous or superfluous information. Students should be able to grasp what the AI is communicating in a short duration.

**Relevance for CS Students**

Students only have a limited time to learn many topics. Being bogged down with irrelevant or superfluous information would make the tool have less utility than simply studying the base materials.

**Concrete Example Interaction**

If the user asks for a summary of topics covered in formal languages, they expect the information provided to give an overview while not delving into unnecessary details or going on tangents.


## 4. Trust and Transparency of AI Responses
**Definition**

Users should be able to see exactly where their information is being sent and how it’s being used (e.g., notifying them of external APIs like OpenAI). Materials the model references should be linked in the response, and users should be informed clearly if errors occur. The fallibility of the model should be acknowledged in the UI and clearly visible to the user.

**Relevance for CS Students**

People tend to trust a system if it has seemed to be reliable, or they will have complete distrust for “black boxes.” Presenting a balance of transparency information to encourage the user to be vigilant, while giving them reasons that the system is working properly, can help solve both problems. If a user is led to trust a system that misinforms them, the system is incredibly flawed.

**Concrete Example Interaction**

A student asks for the definition of bubble sort. The system gives a faulty explanation but also shows the explanation from course content directly. The student is able to spot the contradiction and learn the correct information while reinforcing their wariness of false information from AI systems.


## 5. Error Handling
**Definition**

Errors should be reported in a manner that is easily understandable to the user. The application outside of external API calls and responses should be error-free. Errors should be handled gracefully and should only be reported to the user if they cannot be solved programmatically.

**Relevance for CS Students**

LLM APIs cannot be trusted to be 100% reliable. From downtime to faulty tool calls, there is a large variety of issues that can be generated by these APIs. In order to give the user the best possible experience, they should be informed of the exact issue so that they can make informed choices on how to proceed.

**Concrete Example Interaction**

A user asks for quiz questions, but the AI responds in a format other than that expected for quiz questions. The program identifies this discrepancy and automatically resubmits the prompt to receive the desired answer. The end user doesn’t have to spend additional time figuring out the problem and submitting a new prompt, which would lead to increased frustration.
